ข้อค้นพบเชิงนโยบายKey Policy Findings

1. การกรองข้อมูลอย่างเป็นระบบ — จากการร้องเรียน 266 จุด เหลือ 127 จุดที่ตรงกับขอบเขต Traffic Engineering และ 77 จุดที่อยู่ในพื้นที่รับผิดชอบ กทม. โดยตรง สะท้อนกระบวนการ refine inventory ที่ขับเคลื่อนด้วยข้อมูล
2. Persistent Hotspots — 23 จุด ปรากฏใน inventory ทั้ง 3 ปี ตามพิกัด (Spatial match ≤ 250m) จุดเหล่านี้ต้องการ การแก้ไขเชิงโครงสร้าง (structural intervention) และความร่วมมือระหว่างหน่วยงานหลายฝ่าย
3. 25 Persistent Corridors ที่จัดกลุ่มด้วย DBSCAN เป็นพื้นที่ปัญหาเชิงเส้น (corridor) เช่น ดุสิต–สามเสน, บางกอกน้อย–อรุณอมรินทร์, ปทุมวัน–พระรามที่ 1, บางบอน–เอกชัย — เหมาะกับการแก้แบบ corridor-level intervention มากกว่าจุดเดี่ยว
4. ความร่วมมือหลายหน่วยงาน — การแก้ไขจริงต้องใช้ความร่วมมือของ สจส. · สนย. · สำนักเทศกิจ · สำนักงานเขต · บช.น. ตามลักษณะ root cause ของแต่ละจุด

วิธีการเชิงสถิติ / Statistical Methods

📐
Haversine Distance Matching
threshold = 250m
จับคู่จุดข้ามปีโดยใช้สูตรระยะทาง Great-Circle บนผิวโลก เพื่อระบุว่าจุดใดสืบทอดจากปีก่อน (carry-over) จุดใดเป็นจุดใหม่ (new) และจุดใดถูกตัดออก (drop-out)
🎯
K-Means Clustering
k=6 · Silhouette = 0.46
จัดกลุ่มจุดปัญหาตามรูปแบบมาตรการที่ใช้ (6 มิติ: บริหารจัดการ · กวดขัน · สัญญาณไฟ · กายภาพ · BusBay · CCTV) เพื่อหา cluster ที่มีอัตราความสำเร็จสูงสุด
🛣️
DBSCAN Corridor Clustering
eps=400m · min_samples=3
จัดกลุ่ม corridor ตามพิกัดจริง (ไม่ขึ้นกับขอบเขตการปกครอง) ด้วย Density-Based Spatial Clustering — ระบุได้ 45 corridors โดย 25 corridors active ทั้ง 3 ปี
🌡️
Kernel Density Estimation
Gaussian · bw=0.05
คำนวณ density ของจุดปัญหาบน 60×60 grid ครอบคลุม กทม. เพื่อระบุ "hot zones" ที่ปัญหากระจุกตัวสูง โดยไม่ยึดติดกับขอบเขตการปกครอง
📊
Inferential Statistics
Spearman · Kruskal-Wallis · χ²
ทดสอบสมมติฐาน — Spearman correlation (จำนวนมาตรการ ↔ ความสำเร็จ, ρ=0.141, p=0.044*) และ Kruskal-Wallis (ความสำเร็จ × โซน, H=19.83, p=0.001**)
🔄
Data Pipeline & ETL
pandas · openpyxl
รวม schema 3 ปี (ต่างกัน) ให้เป็นรูปแบบเดียวด้วยการ map columns, normalize zones/districts, จัดการ missing values และ export เป็น JSON/GeoJSON สำหรับ dashboard
